## 1. 项目一句话说明

> 交易教练是一款微信小程序，用户每天花 5 分钟记录交易操作，AI 教练"严正"会结合历史复盘数据，分析行为偏差、识别认知陷阱、给出可执行的改进建议。核心理念是：**交易是可以训练的技能，复盘是最好的训练方式。**

面试时用这句话开场，30 秒让面试官理解你在做什么。

## 2. 需求与产品理解

### 2.1 解决什么问题

个人交易者面临一个普遍困境：**知道自己应该复盘，但做不到。**

原因有三：
1. **不知道怎么复盘** — 面对空白文档无从下手，写出来的都是流水账
2. **坚持不下来** — 没有正反馈，复盘变成负担
3. **复盘没有闭环** — 写了就忘，同样的错误反复犯

### 2.2 产品方案

| 问题 | 解决方案 | 对应功能 |
|------|---------|---------|
| 不知道怎么复盘 | 结构化表单引导 | 市场分析 → 知行合一 → 自我评价 → 明日计划 |
| 坚持不下来 | AI 即时反馈 + 数据可视化 | 提交后 30 秒内得到教练点评；仪表盘追踪进步 |
| 复盘没有闭环 | 记忆闭环机制 | AI 追问未回答的问题；昨日计划今日追踪 |

### 2.3 目标用户画像

- A 股个人交易者，有一定交易经验
- 知道复盘重要但缺乏系统方法
- 愿意每天花 5-10 分钟记录
- 对 AI 辅助分析持开放态度

### 2.4 核心用户旅程

```
首次使用：
  打开小程序 → 看到欢迎引导 → 填写第一笔交易 → 提交
  → 30 秒后看到 AI 教练的深度分析 → "原来我有这个问题"
  → 决定明天继续复盘

日常使用：
  收盘后打开 → 市场数据自动预填 → 记录今日操作
  → 昨日计划自动弹出，标记执行情况 → 提交
  → AI 教练结合历史数据分析 → 查看仪表盘趋势
```

### 2.5 产品结构（8 个页面）

```
TabBar（4 页）：
  复盘（index）     — 核心入口，结构化表单
  数据（dashboard） — 数据仪表盘，可视化指标
  自选（stocks）    — 自选股管理，实时行情
  我的（profile）   — 设置、数据管理、主题切换

非 TabBar（4 页）：
  AI 教练（coach）       — AI 分析结果展示（流式输出）
  复盘详情（history）    — 单条复盘查看/编辑/删除
  周期复盘（period）     — 周/月维度 AI 聚合分析
  对话历史（conversation） — AI 对话记录管理
```

### 2.6 数据流（面试必讲）

```
用户填写表单
    │
    ├─ 每 500ms 自动保存草稿（防丢失）
    │
    ▼
点击「提交复盘」
    │
    ├─ 表单序列化为 XML 格式（escapeXml 防注入）
    ├─ 携带 XML + JSON 跳转 coach 页面
    │
    ▼
coach 页面 onLoad
    │
    ├─ 解析 formData（try-catch 保护）
    ├─ 创建对话记录，保存到 Storage
    ├─ 并行请求：历史 5 条复盘 + 实时行情
    │
    ▼
组装 Prompt
    │
    ├─ System Prompt（严正教练人设 + 分析框架 + 输出结构）
    ├─ User Message = 复盘 XML + 历史上下文 + 行情数据 + 未回答追问
    │
    ▼
调用 DeepSeek API（temperature=0.8）
    │
    ▼
流式打字机效果展示（20ms/3 字符）
    │
    ├─ 同时：解析 __TAGS__ 提取行为标签
    ├─ 同时：解析「必须回答的问题」段落，存为 pendingQuestions
    ├─ 保存复盘记录 + 更新对话记录 + 清除草稿
    │
    ▼
仪表盘实时更新（computeAllStats 从 Storage 读取计算）
```

**面试时讲这个流程，能展示你对数据流、异步处理、状态管理的理解。**

## 3. 技术架构详解

### 3.1 整体架构

```
┌───────────────────────────────────────────────────────┐
│                     微信小程序客户端                     │
├──────────────┬────────────────┬───────────────────────┤
│    页面层      │    组件层       │      工具层            │
│              │                │                       │
│  index       │  stat-card     │  storage.js  (数据持久化) │
│  dashboard   │  mini-chart    │  api.js      (AI 调用)   │
│  stocks      │  tag-cloud     │  market.js   (行情拉取)   │
│  profile     │  progress-bar  │  stats.js    (统计引擎)   │
│  coach       │                │  theme.js    (主题管理)   │
│  history     │                │  stockSearch (股票搜索)   │
│  period      │                │  export.js   (数据导出)   │
│  conversation│                │  date.js     (日期工具)   │
├──────────────┴────────────────┴───────────────────────┤
│              设计系统（CSS 变量 + 通用样式）               │
│  --primary: #D4574E  --bg: #F8F5F2  --card-bg: #fff   │
├───────────────────────────────────────────────────────┤
│              微信原生 API                                │
│  wx.setStorageSync / wx.request / wx.navigateTo        │
├───────────────────────────────────────────────────────┤
│           外部服务                                      │
│  DeepSeek API (AI)  东方财富 API (行情)                  │
└───────────────────────────────────────────────────────┘
```

### 3.2 技术选型与理由

| 选型 | 理由 |
|------|------|
| 微信小程序原生框架 | 无构建工具、无 npm 依赖，开发效率高，直接在微信开发者工具中编译运行 |
| DeepSeek API | 国产大模型，中文理解能力强，API 兼容 OpenAI 格式，价格低 |
| 东方财富公开 API | 免费、无需注册、数据覆盖 A 股全量，适合个人项目 |
| 本地 Storage | 无后端依赖，数据完全在用户设备上，隐私安全 |
| CSS 变量设计系统 | 一套代码支持亮色/暗色主题，修改变量即全局生效 |

### 3.3 设计系统（面试可讲）

采用**三层 CSS 变量架构**：

```css
/* 第一层：基础变量（原始值） */
--color-red-500: #D4574E;

/* 第二层：语义变量（用途） */
--primary: var(--color-red-500);

/* 第三层：组件变量 */
--button-bg: var(--primary);
```

实际实现中简化为两层（基础 + 语义），因为项目规模不需要三层。

**设计规范：**
- 主色 `#D4574E`（柔和红，A 股红涨寓意）
- 底色 `#F8F5F2`（暖灰白，减少视觉疲劳）
- 圆角统一 20rpx
- 阴影 `0 2rpx 12rpx rgba(45,42,38,0.05)`
- 渐变用于按钮和重点卡片
- 动画用于交互反馈和状态变化

**数据：** 449 处 CSS 变量引用、45 处渐变、24 处动画、50 处过渡效果。

### 3.4 自定义组件设计

4 个可复用组件，全部通过 properties 接收数据，通过 events 向外通信：

| 组件 | 职责 | 关键 Properties |
|------|------|----------------|
| stat-card | 统计指标卡片 | icon, value, unit, label, badge, trend |
| mini-chart | 迷你图表 | type(bar/trend/ring), chartData, title |
| tag-cloud | 行为标签云 | data(对象，自动计算大小和颜色等级) |
| progress-bar | 进度条 | label, percent, color |

**组件设计原则：**
- 单一职责：每个组件只做一件事
- 数据下沉：通过 properties 接收数据，不直接读 Storage
- 事件上浮：通过 triggerEvent 向父组件通信
- 样式隔离：使用 CSS 变量引用全局设计系统

## 4. 核心模块逐个讲

### 4.1 统计引擎（stats.js）

**职责：** 从本地复盘数据实时计算 15+ 个交易指标。

**核心函数：**
```javascript
computeAllStats(reviews) → {
  totalDays, totalTrades, buyCount, sellCount,
  missedCount, planAdherenceRate, avgTradesPerDay,
  streakDays, tagDistribution, weeklyActivity,
  monthlyTrend, topStocks, winRate, recentScore,
  adherenceTrend, mostCommonMissed, reviewTimeDistribution
}
```

**设计亮点：**
- 纯函数设计：传入 reviews 数组，返回统计对象，无副作用
- 防御性编程：reviews 为空时返回 getEmptyStats() 而非 null
- 单次遍历优化：多个指标在同一个 reduce 循环中计算

**面试可讲的点：**
- 为什么用纯函数？（可测试、可缓存、无副作用）
- 如何处理空数据？（返回合理的默认值，而非 null/undefined）
- 性能考虑？（数据量小时无需优化，数据量大时可加 memoization）

### 4.2 AI 调用模块（api.js + coach.js）

**api.js 职责：** 封装 wx.request 调用 DeepSeek API。

```javascript
callAI(messages, options) → Promise<string>
```

- 支持 temperature、max_tokens、timeout 配置
- 错误分类：超时、网络失败、API 异常
- 返回纯文本（非流式，流式需要后端代理）

**coach.js 职责：** 组装 Prompt、调用 API、处理响应。

**Prompt 工程（核心竞争力）：**

System Prompt 定义了：
1. **人设**："严正"——严肃直击要害的交易教练
2. **分析框架**：5 步分析法（行为对比 → 未执行计划深挖 → 知行合一审计 → 认知偏差诊断 → 市场匹配度）
3. **输出结构**：对比诊断 → 操作审计 → 认知陷阱 → 必须回答的问题 → 改进清单 → 今日金句
4. **标签提取**：`__TAGS__:["标签1","标签2"]` 格式，正则解析

**纵向对比机制：**
- 自动拉取最近 5 条复盘，注入 User Message
- 未回答的追问也会注入，形成"记忆闭环"
- 实时行情数据注入，让 AI 能判断"市场匹配度"

**面试可讲的点：**
- Prompt Engineering 的设计思路
- 为什么用 XML 格式传数据？（结构化、可解析、防注入）
- 如何实现"记忆闭环"？（pendingQuestions 机制）
- 打字机效果的实现（setTimeout 递归 + chunkSize 控制速度）

### 4.3 行情模块（market.js）

**职责：** 从东方财富 API 拉取 A 股实时行情。

**数据源：**
- 上证指数：`secid=1.000001`
- 深证指数：`secid=0.399001`
- 涨跌停统计：全量 A 股遍历，pct >= 9.8 为涨停，<= -9.8 为跌停
- 板块排行：行业板块按涨幅排序取 Top 5

**容错设计：**
- 每个 API 调用独立 catch，失败返回空数据而非阻塞主流程
- 行情快照缓存到 Storage，用于对比昨日成交量
- 全部失败时静默降级，不影响复盘流程

**面试可讲的点：**
- Promise.all 并行请求 vs 串行
- 容错策略：部分失败不影响整体
- 数据缓存策略

### 4.4 存储模块（storage.js）

**职责：** 封装 wx.setStorageSync / wx.getStorageSync，提供类型安全的 CRUD。

**存储 Key：**
- `reviews`：复盘记录数组
- `periodReviews`：周期复盘数组
- `conversations`：对话数组
- `draft`：当前草稿（单条）
- `market_snapshot`：行情快照缓存
- `watchlist`：自选股列表
- `app_theme`：主题偏好

**设计模式：** Repository Pattern — 所有数据访问通过 storage.js，页面不直接调用 wx.setStorageSync。

**面试可讲的点：**
- 为什么封装一层？（统一数据访问、便于后续替换为云存储）
- 合并模式导入（importBackup 用 Set 去重，保留现有数据）
- 存储上限监控（getStorageUsage 返回 currentSize / limitSize）

### 4.5 行为标签系统

**提取机制：**
1. AI 回复末尾输出 `__TAGS__:["追涨","犹豫不决"]`
2. 正则提取：`/__TAGS__\s*:\s*\[(.*?)\]/`
3. JSON.parse 解析为数组
4. 存入 review.tags

**候选标签池（18 种）：**
```
追涨、杀跌、止损拖延、止盈过早、过度交易、犹豫不决、
锚定效应、损失厌恶、过度自信、确认偏误、近因效应、
羊群效应、计划缺失、逆势操作、仓位失控、盘中冲动、
报复交易、踏空焦虑
```

**统计与可视化：**
- computeTagDistribution：按频次降序取 Top 10
- tag-cloud 组件：根据频次自动计算大小和颜色等级
- 仪表盘行为分析 Tab：标签云 + 月度趋势 + 常见遗漏计划

## 5. AI 协作开发过程

### 5.1 协作模式

这个项目是**人机协作开发**的典型案例。我的角色是产品经理 + 架构师，AI（Claude）的角色是全栈开发 + 代码审查。

**分工原则：**
- 我负责：需求定义、产品设计、架构决策、质量把关
- AI 负责：代码实现、样式编写、Bug 修复、文档生成

### 5.2 协作流程

```
第一阶段：需求与架构
  我：定义产品需求、页面结构、数据流
  AI：生成项目骨架、工具函数、组件框架

第二阶段：核心功能开发
  我：描述每个页面的功能需求和交互逻辑
  AI：实现 JS 逻辑、WXML 模板、WXSS 样式
  我：Review 代码，提出修改意见
  AI：迭代修改

第三阶段：UI 打磨
  我：描述设计方向（淡雅、暖色、圆角）
  AI：生成设计系统（CSS 变量）、重写所有页面样式
  我：在微信开发者工具中预览，反馈调整
  AI：迭代优化

第四阶段：质量保障
  AI：全面代码审计，发现安全性、健壮性、性能问题
  我：确认问题，要求修复
  AI：修复并提交
```

### 5.3 AI 协作的关键技巧

1. **先定义再实现** — 先写 PRD 和设计规范，再让 AI 写代码。避免 AI 自由发挥导致风格不一致。

2. **用 Skill 系统提效** — 安装了 `brainstorming`、`writing-plans`、`design-system` 等 Skill，让 AI 遵循结构化的开发流程。

3. **并行 Agent 提速** — 对独立的页面样式重写，启动多个 Agent 并行处理，大幅缩短开发时间。

4. **持续 Review** — 每次 AI 输出后，我在微信开发者工具中实际预览，发现问题立即反馈。

5. **让 AI 审计自己** — 最后让 AI 以"审查者"视角重新审视整个代码库，发现人机都可能遗漏的问题。

### 5.4 协作产出

| 指标 | 数据 |
|------|------|
| 总代码行数 | ~6000 行（JS + WXML + WXSS + JSON） |
| 页面数 | 8 个 |
| 组件数 | 4 个 |
| 工具模块数 | 9 个 |
| AI 参与比例 | ~90% 的代码由 AI 生成 |
| 人工介入 | 需求定义、架构决策、质量把关、真机测试 |

## 6. 四维度分析

### 6.1 产品角度

**做得好的：**
- 结构化表单解决了"不知道怎么复盘"的问题
- AI 即时反馈提供了正循环，激励用户坚持
- 数据仪表盘让进步可视化
- 昨日计划追踪形成了闭环

**可以改进的：**
- 缺少社交功能（分享到朋友圈、排行榜）
- 没有提醒功能（收盘后推送复盘提醒）
- 没有盈亏记录（只有"是否符合计划"，没有实际盈亏数据）
- 没有新手引导流程（首次使用只有静态欢迎卡片）

**面试话术：**
> "这个项目的 MVP 聚焦在'复盘 → AI 分析 → 数据追踪'这条核心链路上。社交、提醒、盈亏记录等功能在验证核心价值后再迭代。这是典型的精益开发思路。"

### 6.2 测试角度

**当前状态：** 没有自动化测试（无测试框架、无测试用例）。

**可测试的模块：**
- `stats.js` — 纯函数，最适合单元测试
- `storage.js` — 可 mock wx.setStorageSync 后测试
- `export.js` — 纯数据转换，可测试
- `date.js` — 纯函数，可测试

**不易测试的模块：**
- 页面 JS — 依赖 wx 和 Page 构造器，需要小程序测试框架
- API 调用 — 依赖 wx.request，需要 mock
- 组件 — 依赖 Component 构造器

**面试话术：**
> "当前版本没有自动化测试，这是下一步的重点改进方向。统计引擎 stats.js 是纯函数设计，天然适合单元测试。页面层的测试需要借助 miniprogram-simulate 等小程序测试框架。"

**如果面试官追问怎么测试：**
```javascript
// stats.js 的测试示例
describe('computePlanAdherence', () => {
  it('should return 0 for empty reviews', () => {
    expect(computePlanAdherence([])).toBe(0)
  })

  it('should return 100 when all trades match plan', () => {
    const reviews = [{
      formData: {
        buyList: [{ stock: 'A', matchPlan: true }],
        sellList: []
      }
    }]
    expect(computePlanAdherence(reviews)).toBe(100)
  })
})
```

### 6.3 开发角度

**架构优点：**
- 清晰的分层：页面 → 组件 → 工具
- Repository Pattern：所有数据访问通过 storage.js
- 设计系统：CSS 变量驱动，一套代码支持主题切换
- 组件化：4 个可复用组件，单一职责

**架构不足：**
- 没有状态管理库（依赖页面 data + Storage，跨页面通信靠 getCurrentPages）
- 没有路由管理（硬编码 URL 字符串）
- 没有构建工具（无法使用 TypeScript、ESLint、Prettier）
- API Key 硬编码在客户端（生产环境需要后端代理）

**面试话术：**
> "原生小程序框架没有官方的状态管理方案，我用 Storage 作为'单一数据源'，页面通过 onShow 生命周期刷新数据。对于这个规模的项目，这种方式够用且简单。如果项目增长到 20+ 页面，我会考虑引入 mobx-miniprogram 做状态管理。"

### 6.4 运维与运营角度

**运维：**
- 纯前端项目，无需服务器运维
- 数据存储在用户设备上，无需数据库
- API 调用直连 DeepSeek，无需后端
- 唯一的运维需求：API Key 轮换（如果泄露）

**运营：**
- 用户增长：依赖口碑传播（微信分享）
- 用户留存：依赖复盘习惯养成（仪表盘数据激励）
- 数据分析：本地存储，无法做服务端埋点
- 反馈收集：无内置反馈入口

**面试话术：**
> "这是一个纯前端架构，运维成本几乎为零。代价是无法做服务端数据分析和用户行为追踪。如果要做商业化，第一步是加后端代理（保护 API Key + 用户认证），第二步是加数据埋点。"

## 7. 面试高频问题与参考回答

### Q1：介绍一下你的项目？

> 这是一款微信小程序，叫"交易教练"。用户每天花 5 分钟记录交易操作，AI 教练会结合历史复盘数据，分析行为偏差、识别认知陷阱、给出改进建议。
>
> 技术上是纯前端架构，用微信小程序原生框架，AI 用 DeepSeek API，行情数据来自东方财富。没有后端，所有数据存储在用户本地。
>
> 我负责了需求定义、产品设计、架构决策，代码部分大部分是和 AI 协作完成的。

### Q2：你用了什么技术栈？为什么选这些？

> 微信小程序原生框架，不用 Taro/uni-app 的原因是：项目不需要跨平台，原生框架没有构建步骤，开发效率最高。
>
> AI 选 DeepSeek 而不是 OpenAI，原因是：中文理解能力更强，API 兼容 OpenAI 格式，价格低。
>
> 行情数据选东方财富而不是同花顺，原因是：东方财富的公开 API 免费、无需注册、数据覆盖全。

### Q3：你的项目有什么亮点？

> 三个亮点：
> 1. **Prompt Engineering** — 设计了一套结构化的 AI 分析框架，包含 5 步分析法和标签提取机制，不是简单的"帮我分析一下"
> 2. **记忆闭环** — AI 提取的追问会注入下一次复盘 prompt，形成连续对话，不是每次独立分析
> 3. **统计引擎** — 纯函数设计的 stats.js，一次遍历计算 15+ 个指标，可测试、无副作用

### Q4：你遇到过什么技术难点？怎么解决的？

> **难点 1：AI 回复的结构化解析。**
> AI 回复是自由文本，我需要从中提取行为标签和追问问题。解决方案是在 System Prompt 中定义输出格式（`__TAGS__` 和 `### 必须回答的问题`），然后用正则提取。
>
> **难点 2：本地存储的数据管理。**
> 所有数据存在 wx.setStorageSync，没有 SQL 查询能力。解决方案是封装 storage.js 作为 Repository 层，所有数据访问统一入口，stats.js 从内存中计算指标。
>
> **难点 3：设计系统的一致性。**
> 8 个页面 + 4 个组件需要统一视觉风格。解决方案是 CSS 变量驱动的设计系统，修改变量即全局生效，449 处变量引用确保一致性。

### Q5：如果让你重新做，你会怎么改进？

> 1. **加后端代理** — 保护 API Key，支持用户认证和数据同步
> 2. **加自动化测试** — stats.js 用 Jest 单元测试，页面用 miniprogram-simulate
> 3. **引入状态管理** — 用 mobx-miniprogram 管理全局状态，减少 getCurrentPages 的使用
> 4. **加盈亏记录** — 不只是"是否符合计划"，还要记录实际盈亏，计算真实的胜率和收益率

### Q6：你的 AI 调用是怎么做的？有没有考虑过成本？

> 用 wx.request 直连 DeepSeek API，每次复盘调用一次，消耗约 1000-2000 token。
>
> 成本方面：DeepSeek 的价格大约是每百万 token 1-2 元，一次复盘成本不到 1 分钱。对于个人项目完全可接受。
>
> 安全方面：API Key 硬编码在客户端，这是已知的安全隐患。生产环境需要后端代理来隐藏 Key。

### Q7：你的 CSS 设计系统是怎么做的？

> 用 CSS 变量实现。在 app.wxss 中定义全局变量（--primary、--bg、--card-bg 等），所有页面和组件通过 var() 引用。
>
> 主题切换时，只需在 page 标签上切换 class（dark-mode），CSS 变量自动切换到暗色值，所有组件同步响应。
>
> 整个项目有 449 处 CSS 变量引用，确保了视觉一致性。

### Q8：你是怎么做数据可视化的？

> 用自定义组件实现，没有引入第三方图表库。
>
> mini-chart 组件支持三种类型：柱状图（bar）、趋势图（trend）、环形图（ring）。实现方式是用 CSS 的 height 百分比和 conic-gradient，不需要 Canvas。
>
> tag-cloud 组件根据标签频次自动计算字体大小和颜色等级。
>
> stat-card 组件支持 icon/value/unit/label/badge/trend 六个属性。
>
> 这种方式的优点是轻量（无依赖）、性能好（纯 CSS）、可定制（完全控制样式）。

### Q9：你的项目有没有做性能优化？

> 几个关键优化：
> 1. **自动保存防抖** — autoSaveDraft 用 500ms 防抖，避免频繁写 Storage
> 2. **行情数据缓存** — market_snapshot 缓存到 Storage，用于对比昨日成交量
> 3. **懒加载** — app.json 配置 lazyCodeLoading: requiredComponents
> 4. **并行请求** — coach 页面用 Promise.all 同时请求历史数据和行情数据
> 5. **组件按需渲染** — wx:if 控制 Tab 内容的渲染，非活跃 Tab 不渲染

### Q10：你和 AI 是怎么协作的？

> 我是产品经理 + 架构师，AI 是全栈开发。
>
> 流程是：我定义需求和设计规范 → AI 生成代码 → 我在微信开发者工具中预览 → 反馈修改 → AI 迭代。
>
> 关键技巧是：先写 PRD 再写代码，用 CSS 变量约束 AI 的设计输出，用多个 Agent 并行处理独立页面。
>
> 最后让 AI 以"审查者"视角审计自己的代码，发现了几个安全性问题（JSON.parse 无 try-catch、定时器未清理等）并修复。
